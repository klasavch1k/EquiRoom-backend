# 📋 Отчёт для фронтенда: Изменения безопасности данных (конкурентность)

> **Дата:** 08.04.2026  
> **Цель:** Защита данных при многопользовательской работе — гонки данных, дубликаты, конкурентное редактирование.

---

## 🔑 ГЛАВНОЕ: Новые форматы ошибок, которые нужно обрабатывать

### Было раньше
Почти все конкурентные конфликты приводили к HTTP `500 Internal Server Error` без полезного сообщения.

### Стало теперь
Все конфликты возвращают **структурированные ошибки**. Фронт должен обрабатывать **два формата** ответов при ошибках:

#### Формат 1 — простой (без code):
```json
{
  "message": "Текст ошибки на русском"
}
```
Используется для: `400 Bad Request`, `403 Forbidden`, `404 Not Found`, `409 Conflict` (от `IllegalStateException`), `422 Unprocessable Entity`

#### Формат 2 — с кодом (code + message):
```json
{
  "code": "VERSION_CONFLICT",
  "message": "Данные были изменены другим пользователем. Обновите страницу."
}
```
Используется для: `409 Conflict` (от оптимистической блокировки, конфликтов данных, read-only шоу, фазы судейства)

---

## 📌 Полный список кодов ошибок (поле `code` в ответе)

| `code` | HTTP-статус | Когда возникает | Что делать на фронте |
|--------|-------------|-----------------|---------------------|
| `VERSION_CONFLICT` | 409 | Два пользователя одновременно редактировали одну сущность (шоу, регистрацию, entry) | Показать тост «Данные устарели», **перезагрузить данные** с сервера (повторный GET) |
| `DATA_CONFLICT` | 409 | Необработанное нарушение уникальности в БД (fallback) | Показать тост «Операция не удалась, попробуйте ещё раз» |
| `SHOW_READ_ONLY` | 409 | Попытка изменить завершённое шоу или шоу после endDate | Показать «Шоу завершено, изменения невозможны», заблокировать UI редактирования |
| `JUDGING_PHASE` | 409 | Попытка поставить оценки вне фазы судейства | Показать «Фаза судейства не активна» |

> ⚠️ **Важно:** Если HTTP `409` пришёл **без поля `code`** — это `IllegalStateException`, смотри поле `message` (оно всегда на русском и содержит понятное описание).

## 🛠 Рекомендация: глобальный обработчик ошибок на фронте

Рекомендую добавить в axios/fetch interceptor глобальную обработку:

```javascript
// Пример для axios
axios.interceptors.response.use(
  response => response,
  error => {
    const status = error.response?.status;
    const data = error.response?.data;

    if (status === 409) {
      if (data?.code === 'VERSION_CONFLICT') {
        // Данные устарели — нужно перезагрузить
        showToast('Данные были изменены. Обновляем страницу...');
        // Перезагрузить данные текущей страницы
        window.location.reload(); // или более тонкая логика
        return Promise.reject(error);
      }
      if (data?.code === 'SHOW_READ_ONLY') {
        showToast('Шоу завершено, изменения невозможны');
        return Promise.reject(error);
      }
      if (data?.code === 'JUDGING_PHASE') {
        showToast('Фаза судейства не активна');
        return Promise.reject(error);
      }
      if (data?.code === 'DATA_CONFLICT') {
        showToast('Конфликт данных. Попробуйте ещё раз.');
        return Promise.reject(error);
      }
      // 409 без code — это IllegalStateException, показываем message
      if (data?.message) {
        showToast(data.message);
        return Promise.reject(error);
      }
    }

    if (status === 403) {
      showToast(data?.message || 'Нет доступа');
      return Promise.reject(error);
    }

    if (status === 400) {
      showToast(data?.message || 'Некорректный запрос');
      return Promise.reject(error);
    }

    return Promise.reject(error);
  }
);
```
