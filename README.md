## О проекте

ModelHorse — это не просто учебный проект. Это попытка закрыть реальную потребность небольшого, но активного сообщества: коллекционеры моделей лошадей до сих пор координируются через разрозненные чаты, таблицы и форумы.

Платформа задумана как единое пространство для трёх аудиторий:

- **Коллекционеры** — ведут каталог своих моделей, отслеживают коллекции друг друга, участвуют в шоу и мероприятиях.
- **Мастера** (художники, реплейнтеры) — управляют заказами и очередью работ, прозрачно показывают прогресс заказчикам, ведут портфолио в контексте хобби.
- **Организаторы шоу** — проводят онлайн-выставки с полноценной судейской системой, управлением регистрацией и структурой классов.

В перспективе — торговая площадка для продаж и обменов, ивенты, розыгрыши. Идея проверена опросами внутри сообщества: участники готовы платить за специализированный инструмент, написанный под их хобби.

**На текущем этапе реализован стартовый модуль:** система проведения шоу и управления коллекцией — функциональный фундамент, с которого начинается онбординг пользователей.

## Архитектура

Приложение разбито на стандартные слои с явным разделением ответственности:

```
controller/   → принимает HTTP-запросы, извлекает Principal из SecurityContext
service/      → вся бизнес-логика, проверки прав, транзакции
repository/   → Spring Data JPA репозитории
entity/       → JPA-сущности (Hibernate)
dto/          → объекты запросов и ответов (отделены от сущностей)
security/     → JWT-фильтр, CustomUserDetails, JwtUtil
exception/    → единый @RestControllerAdvice с типизированными ошибками
config/       → SecurityConfig, WebConfig
```

Контроллер не обращается к репозиторию напрямую — только через сервис. Исключение сделано лишь для быстрых exists-проверок в нескольких эндпоинтах (помечено к рефакторингу).

## Функциональность

### Пользователи
- Регистрация и вход с выдачей JWT-токена
- Профиль с аватаром, система подписок (follow/unfollow)
- Роли: `USER`, `MASTER`, `ORGANIZER`, `JUDGE` можно иметь несколько ролей одновременно

### Коллекция моделей
- CRUD для моделей лошадей: название, порода, масть, автор, год покраски
- Загрузка медиафайлов (фото/видео) через multipart
- Информация о продаже, привязка наград

### Шоу
- Создание и редактирование шоу с баннером и файлом регламента
- Настройка платности, лотереи, лимита дополнительных фото
- Управление организаторами (co-organizer) и судьями
- Тикет-система: разные типы билетов с ценами
- Жизненный цикл: открытие регистрации → начало судейства → завершение

### Структура шоу
- Иерархия: **Дивизион → Секция → Класс**
- Настройка критериев оценивания для дивизиона (название критерия + максимальный балл)
- Критерии блокируются после старта шоу

### Регистрация и записи (Entry)
- Регистрация участника на шоу с выбором тикета и спонсорства(обсуждается с организаторами)
- Подача моделей в конкретный класс (entry) с фотографиями
- Статусы регистрации: `PENDING`, `APPROVED`, `REJECTED`

### Судейство
- Страница судьи: список entry в классе с уже выставленными оценками
- Выставление и обновление баллов по каждому критерию
- Доступ к судейству строго в рамках активной фазы шоу

### Права доступа
Несколько уровней в контексте шоу: **организатор-создатель → организатор → судья → участник → гость**. Проверки реализованы в сервисном слое через явные методы `isOrganizer()`, `isCreator()`, `isJudge()`.

## Безопасность

### Аутентификация
- Stateless JWT-аутентификация: токен содержит `userId` и список ролей
- `JwtFilter` перехватывает запросы и устанавливает `SecurityContext`
- Пароли хешируются BCrypt

### Защита от конкурентных конфликтов
В многопользовательских сценариях (несколько организаторов редактируют шоу одновременно) применяется **оптимистичная блокировка** через `@Version` на сущностях `Show`, `Registration`, `Entry`.

При конфликте клиент получает структурированный ответ:

```json
{
  "code": "VERSION_CONFLICT",
  "message": "Данные были изменены другим пользователем. Обновите страницу."
}
```

Все ошибки обрабатываются централизованно в `ApiExceptionHandler`:

| Код ошибки | HTTP | Причина |
|---|---|---|
| `VERSION_CONFLICT` | 409 | Оптимистичный конфликт версий |
| `SHOW_READ_ONLY` | 409 | Попытка изменить завершённое шоу |
| `JUDGING_PHASE` | 409 | Оценки вне активной фазы судейства |
| `DATA_CONFLICT` | 409 | Нарушение уникальности в БД (fallback) |

Дублирующие регистрации защищены частичным уникальным индексом на уровне БД (Flyway-миграция V2).

## Запуск

### Требования
- Java 21
- PostgreSQL 15+
- Docker (опционально)

### Локально

```bash
# 1. Создать базу данных
createdb modelhorse

# 2. Задать переменные окружения
export JWT_SECRET=your_secret_here

# 3. Запустить
./gradlew bootRun
```

Приложение поднимается на `http://localhost:8080`.  
Flyway автоматически применит миграции при старте.

### Docker

```bash
docker build -t model-horse .
docker run -p 8080:8080 \
  -e SPRING_DATASOURCE_URL=jdbc:postgresql://host.docker.internal:5432/modelhorse \
  -e SPRING_DATASOURCE_USERNAME=postgres \
  -e SPRING_DATASOURCE_PASSWORD=your_pass \
  -e JWT_SECRET=your_secret \
  model-horse
```
